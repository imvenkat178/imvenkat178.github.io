Introduction to Distributed Systems

What is a Distributed System?

A distributed system is a collection of independent computers that appear to its users as a single coherent system. These computers communicate and coordinate their actions by passing messages to one another. Distributed systems offer several benefits, such as improved performance, scalability, and fault tolerance. They are essential for handling large-scale applications that require high availability and reliability.

Why Use Distributed Systems?

Distributed systems are used to improve performance, achieve scalability, and ensure fault tolerance. By distributing the workload across multiple machines, we can handle more data and users than a single machine could manage alone. This improves performance and ensures the system can scale to meet demand.

Key Terms in Distributed Systems

Horizontal Scaling

Horizontal scaling, also known as scaling out, involves adding more machines to a system to handle increased load. This method provides flexibility and can handle a large number of requests efficiently. For example, a web application might add more servers to distribute traffic.

In a real-world scenario, an e-commerce website experiencing a surge in traffic during a sale can add more servers to balance the load. However, if not managed correctly, horizontal scaling can lead to complexity in data synchronization and consistency across servers.

Vertical Scaling

Vertical scaling, or scaling up, involves adding more power (CPU, RAM, etc.) to an existing machine. While this can handle increased load, it has limitations and can be expensive compared to horizontal scaling. Vertical scaling is often used when the application demands more resources than what is currently available on a single server.

For instance, a database server handling complex queries might be upgraded with additional RAM and CPUs. The pitfall here is that there is a limit to how much you can scale vertically, and it can become cost-prohibitive.

Partitions

Partitions divide data into manageable chunks, allowing distributed systems to handle large datasets efficiently. For example, a database might partition data by user ID to distribute the load across multiple servers. This approach improves performance and manageability.

In a real-world scenario, a social media platform might partition user data to ensure that each server handles a subset of users, reducing the load on any single server. However, if the main database fails and there is no partitioning, the system may struggle to handle the load, leading to delays or data loss.

Replicas

Replicas are copies of data stored on multiple machines to ensure data availability and fault tolerance. For example, a distributed database might replicate data across different geographical locations to ensure access even if one location fails.

Consider a cloud storage service that replicates user files across multiple data centers. If one data center goes down, users can still access their files from another location. The challenge is to keep replicas synchronized, and inconsistencies can arise if synchronization fails.

Sharding

Sharding is a specific type of partitioning where data is split across different databases or servers. Each shard contains a portion of the data. For example, an e-commerce platform might shard its product catalog to improve search performance and scalability.

In a real-world scenario, a large online retailer might shard its inventory database by product category, ensuring faster search and retrieval. However, managing shards can become complex, especially when rebalancing data across shards or handling cross-shard queries.

Fault Tolerance

Fault tolerance is the ability of a system to continue operating properly in the event of the failure of some of its components. For example, in a distributed storage system, if one server fails, the system can still access data from another server.

In a banking system, fault tolerance ensures that transactions are processed reliably even if a server goes down. The pitfall here is the increased complexity and overhead to implement fault tolerance, which can impact performance if not optimized correctly.

Load Balancing

Load balancing distributes incoming network traffic across multiple servers to ensure no single server becomes overwhelmed. For instance, a load balancer can distribute web traffic among several web servers to optimize resource use and improve response times.

In a real-world example, an online streaming service like Netflix uses load balancing to distribute user requests across multiple servers, ensuring smooth playback without buffering. However, if the load balancer fails, it can create a single point of failure, disrupting the service.
